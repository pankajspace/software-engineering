## Timers
1. [Timers](https://nodejs.org/api/timers.html)

The setImmediate(), setInterval(), and setTimeout() methods each return objects that represent the scheduled timers. These can be used to cancel the timer and prevent it from triggering.

```js
const timeoutObj = setTimeout(() => {
  console.log('timeout beyond time');
}, 1500);

const immediateObj = setImmediate(() => {
  console.log('immediately executing immediate');
});

const intervalObj = setInterval(() => {
  console.log('interviewing the interval');
}, 500);

clearTimeout(timeoutObj);
clearImmediate(immediateObj);
clearInterval(intervalObj);
```

For the promisified variants of setImmediate() and setTimeout(), an AbortController may be used to cancel the timer. When canceled, the returned Promises will be rejected with an 'AbortError'.

```js
// setImmediate example:
const { setImmediate: setImmediatePromise } = require('node:timers/promises');

const ac = new AbortController();
const signal = ac.signal;

setImmediatePromise('foobar', { signal })
  .then(console.log)
  .catch((err) => {
    if (err.name === 'AbortError')
      console.error('The immediate was aborted');
  });

ac.abort();
```

```js
// setTimeout example:
const { setTimeout: setTimeoutPromise } = require('node:timers/promises');

const ac = new AbortController();
const signal = ac.signal;

setTimeoutPromise(1000, 'foobar', { signal })
  .then(console.log)
  .catch((err) => {
    if (err.name === 'AbortError')
      console.error('The timeout was aborted');
  });

ac.abort();
```

## Difference between package.json and package-lock.json
1. [package.json](https://docs.npmjs.com/cli/v10/configuring-npm/package-json)
2. [package-lock.json](https://docs.npmjs.com/cli/v10/configuring-npm/package-lock-json)

In a Node.js project, the package.json file is mandatory, while the package-lock.json file is not mandatory but highly recommended for projects using npm for package management.

package.json is a manifest file that contains metadata about the project and specifies its dependencies. It defines the project configuration. It includes details such as the project name, version, entry point, script commands, and most importantly, the list of dependencies and their versions.

package.lock.json is automatically generated by npm, when dependencies are installed or updated. It contains a detailed record of the exact versions of all dependencies that were installed for the project at a specific point in time. this file is crucial for enabling reproducible and consistent builds in a project.

## Difference between pipe and chaining in a node?
### Pipe: 
In Node.js, the “pipe” method is used to direct the output of one stream to the input of another stream. It is commonly used with readable and writable streams to efficiently transfer data from one source to another without explicitly having to manage the data flow.
```js
const fs = require('fs');

const readableStream = fs.createReadStream('input.txt');
const writableStream = fs.createWriteStream('output.txt');

readableStream.pipe(writableStream);
```

### Chaining: 
Chaining in Node.js generally refers to method chaining, which is the practice of calling multiple methods on an object in a single statement, with each method returning the object itself, thereby allowing for a sequence of operations to be performed on the object in a linear fashion.
```js
const result = [1, 2, 3, 4, 5]
  .map(num => num * 2)
  .filter(num => num > 5)
  .reduce((acc, num) => acc + num, 0);

console.log(result); // Output: 24
```
Both concepts are fundamental to working with streams and asynchronous operations in Node.js.

## Difference between req.params, req.query and req.body in express?
In the context of a Node.js server using the Express framework, req.params and req.query are used to access different types of parameters passed in the URL.

### req.params 
This is an object containing properties mapped to the named route parameters. These parameters are part of the URL path and are matched by the route’s path pattern. They are typically used to capture dynamic values from the URL.
  
```js
// Route definition
app.get('/users/:id', (req, res) => {
  const userId = req.params.id; // Access the "id" parameter from the URL
  // Use userId to retrieve user data
});
```

### req.query 
This is an object containing a property for each query string parameter in the route. Query parameters are appended to the URL after a ? and are used to pass additional information or filters for the requested resource.
If you have a route defined as /search, and a client makes a request like /search?city=NewYork&active=true, you can access the query parameters city and active using req.query.city and req.query.active respectively.

```js
// Route definition
app.get('/search', (req, res) => {
  const city = req.query.city; // Access the "city" query parameter
  const active = req.query.active; // Access the "active" query parameter
  // Use city and active for searching
});
```

### req.body
This is an object containing the parsed request body. It is typically used with POST requests where data is sent in the request body, such as form submissions or API calls. To access the request body, you need to use a body parsing middleware like body-parser or express.json().

```js
// Route definition
app.post('/users', (req, res) => {
  const userData = req.body; // Access the request body
  // Process the user data
});
```
In summary, req.params is used to access route parameters, req.query is used to access query string parameters, and req.body is used to access the request body in an Express application.

## How to read, write, and compress a file in node?
To read a file in Node.js, you can use the fs module’s readFile function.
```js
const fs = require('fs');

fs.readFile('input.txt', 'utf8', (err, data) => {
  if (err) {
    console.error('Error reading the file:', err);
    return;
  }
  console.log('File content:', data);
});
```

To write data to a file in Node.js, you can use the fs module’s writeFile function.
```js
const fs = require('fs');

const data = 'This is the data to write to the file';

fs.writeFile('output.txt', data, (err) => {
  if (err) {
    console.error('Error writing to the file:', err);
    return;
  }
  console.log('Data has been written to the file');
});
```

To compress a file in Node.js, you can use the zlib module to create a GZIP archive.
```js
const fs = require('fs');
const zlib = require('zlib');

const readStream = fs.createReadStream('input.txt');
const writeStream = fs.createWriteStream('input.txt.gz');
const gzip = zlib.createGzip();

readStream.pipe(gzip).pipe(writeStream);

writeStream.on('finish', () => {
  console.log('File has been compressed');
});
```

## What is cron job?
A cron job is a time-based job scheduler. It allows users to schedule tasks (commands or scripts) to run periodically at fixed times, dates, or intervals.

In Node.js, you can use the node-cron module to schedule jobs to run at specific times.

```js
const cron = require('node-cron');

// Schedule a job to run every minute
cron.schedule('* * * * *', () => {
  console.log('Running scheduled job every minute');
});
```
The first argument ‘* * * * *’ represents the cron expression for running the job every minute. The second argument … is a function that will be executed when the scheduled time is reached.

Run a Job Every Hour:
```js
const cron = require('node-cron');

// Schedule a job to run every hour
cron.schedule('0 * * * *', () => {
  console.log('Running scheduled job every hour');
});
```

Run a Job Every Day at a Specific Time:
```js
const cron = require('node-cron');

// Schedule a job to run at 8:00 AM every day
cron.schedule('0 8 * * *', () => {
  console.log('Running scheduled job at 8:00 AM every day');
});
```

## How to perform a load test of the node app?
Loadtest is a Node.js module that allows you to perform load testing on HTTP endpoints. It provides a simple command-line interface for quick testing and also allows for programmatic usage for more complex scenarios.

You can use loadtest to perform a simple load test.
```js
loadtest -n 100 -k 10 http://yourwebsite.com
```

In this command:
1. -n specifies the number of requests to send.
2. -k specifies the number of HTTP keep-alive connections to use.

You can also use loadtest programmatically in your Node.js application. Here’s an example of how you can perform a simple programmatic load test:
```js
const loadtest = require('loadtest');

const options = {
  url: 'http://yourwebsite.com',
  maxRequests: 100,
  concurrency: 10,
};

loadtest.loadTest(options, function (error, ... result) {
  if (error) {
    console.error('Load Test Error: ', error);
  } else {
    console.log('Tests run successfully');
  }
});
```

## When we create a new process in NodeJS will it create a separate host or it will run in the same host like 3000?
When you create a new process in **Node.js**, whether it runs on the same host (e.g., listening on the same port like 3000) or on a separate host/port depends on **how you configure the process**. Here's a breakdown:

### **1. By Default: Same Host, Separate Process**
- When you create a new process in Node.js (e.g., using the **Cluster Module** or `child_process`), it runs on the **same machine (host)** but as a **separate process**.
- Processes share the underlying **CPU** and **RAM** but operate independently in terms of execution.

### **2. Ports in Node.js Processes**
#### **a. Cluster Module**:
- The **Cluster Module** allows multiple worker processes to share the **same port**.
- The **master process** listens on the specified port (e.g., 3000) and distributes incoming connections to worker processes.
- All workers appear to be running on the same port because the load balancing is handled internally by the operating system and the Cluster Module.

#### Example:
If you use `cluster` to scale a server:
- **Host**: Remains the same.
- **Port**: Shared among all worker processes (e.g., 3000).

#### **b. Child Processes**:
- If you create a child process using the `child_process` module and the child process creates its own server, it needs to bind to a different port explicitly.
- **Host**: Remains the same unless explicitly changed.
- **Port**: You must specify a different port for each child server, as two processes cannot listen on the same port.

### **3. Scenarios**

#### **Case 1: Cluster Module**
- All processes share the same **host** and **port**.
- Useful for scaling Node.js applications across CPU cores.
- Example: Multiple workers listening on port 3000.

```plaintext
Host: 127.0.0.1
Port: 3000 (shared by workers)
```

#### **Case 2: Child Processes with Different Ports**
- Each process can listen on a **different port** if configured.
- Useful when you want to run completely independent servers within the same application.

```plaintext
Process 1: Host: 127.0.0.1, Port: 3000
Process 2: Host: 127.0.0.1, Port: 3001
```

#### **Case 3: Same Host, Different Tasks**
- If the child processes don’t open servers but perform other tasks (e.g., computations), they don’t need ports.

### **Key Considerations**
1. **Same Host**:
   - All processes run on the same host unless explicitly configured to use a remote server.

2. **Port Sharing**:
   - Processes can share a port (e.g., via `cluster`) or use separate ports (e.g., with `child_process`).

3. **Network Binding**:
   - A process listens on a specific **host IP** and **port**. By default, Node.js binds to `127.0.0.1` (localhost) unless you specify otherwise.

### **Conclusion**
- **Cluster Module**: Workers share the same port (e.g., 3000) on the same host for load-balanced applications.
- **Child Processes**: Need separate ports unless performing tasks that don’t involve servers.
- Processes always run on the **same host** unless configured otherwise.
